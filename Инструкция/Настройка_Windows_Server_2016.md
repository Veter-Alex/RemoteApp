# Настройка Windows Server 2016 для RemoteApp (Obsidian Portable)

**Назначение:** пошаговая настройка сервера в **рабочей группе** (без домена Active Directory), **без доступа в интернет**, для запуска **Obsidian Portable** с клиентов Windows 7 по **RDP** (сеанс рабочего стола или **удалённый запуск через `.rdp`** — **шаг 5**; **классический RemoteApp** из коллекций RDS обычно требует **домен** и **Connection Broker** — см. **п. 3.7**).

**Что перенести на сервер (с машины с интернетом):**

- всю папку `ObsidianPortable` из корня этого проекта (структуру не менять);
- папку `Инструкция` целиком (этот файл).

**Рекомендуемое расположение на сервере (можно изменить, но тогда править пути в публикации RemoteApp):**

`D:\RemoteApp\ObsidianPortable`  
`D:\RemoteApp\Инструкция`

Отдельный том (`D:\…`) предпочтительнее: меньше сюрпризов с правами, чем у каталога в **корне диска `C:\`** (например `C:\ObsidianPortable`), куда у обычных пользователей по умолчанию часто **нет права записи** — тогда Obsidian падает с ошибкой **`EPERM: operation not permitted`** при открытии `…\Data\ObsidianAppData\obsidian.log`, а **«под администратором»** внезапно работает (повышенные права обходят запрет).

## Юридические и лицензионные особенности

- Роли **Remote Desktop Services** (узел сеансов, RemoteApp) требуют **лицензий RDS CAL** по правилам Microsoft. Без них развёртывание в эксплуатации считается нелицензированным.
- Лицензии **Windows Server** и **клиентов**, подключающихся к RDS, должны соответствовать вашему сценарию.
- **Льготный период RDS (около 120 дней):** после установки **узла сеансов** Microsoft даёт ограниченный срок, в течение которого можно настроить среду **без выданных RDS CAL** (по документации — **120 дней**; точный отсчёт и сообщения на экране см. в [документации Microsoft по лицензированию RDS](https://learn.microsoft.com/windows-server/remote/remote-desktop-services/rds-client-access-license)). По истечении периода, если **не** настроены сервер лицензирования и выдача CAL, **пользовательские** подключения к узлу сеансов обычно **перестают устанавливаться** (ошибка о завершении льготного периода / ненастроенном режиме лицензирования). Это не то же самое, что **два административных** RDP-подключения к серверу без RDS для администрирования — для сценария RemoteApp нужны **RDS User/Device CAL** для постоянной работы. Роль **«Лицензирование»** сама по себе не заменяет покупку пакетов CAL.
- **Рабочая группа (без домена):** в [руководстве по установке узла сеансов без роли Connection Broker](https://learn.microsoft.com/troubleshoot/windows-server/remote/install-rds-host-role-service-without-connection-broker) Microsoft указывает, что в такой конфигурации **нельзя** использовать **режим лицензирования «на пользователя»** — только **на устройство** (*per-device*). Там же сказано, что **инструменты Диспетчера серверов для RDS в рабочей группе не работают** — настройку ролей выполняют через **Windows PowerShell**.

## Обозначения в тексте

| Обозначение        | Смысл                                      |
|--------------------|--------------------------------------------|
| `[ИМЯ_СЕРВЕРА]`    | NetBIOS-имя компьютера (например `RDS01`)  |
| `[IP_СЕРВЕРА]`     | Статический IPv4 сервера в ЛВС             |
| `[ПОЛЬЗОВАТЕЛЬ]`   | Имя локальной учётной записи для входа в RDS (например `obsidian`) |

Заменяйте квадратные скобки на свои значения и везде используйте одни и те же.

## Шаг 0. Первичная проверка после установки ОС

1. Войдите под учётной записью с правами **Администратор** (встроенный `Администратор` или ваша учётка в группе «Администраторы»).
2. Убедитесь, что сервер в **рабочей группе**:  
   **Панель управления** → **Система** → раздел «Имя компьютера, имя домена и параметры рабочей группы» → при необходимости **Изменить параметры** → «Рабочая группа», не домен.
3. Задайте **понятное имя компьютера** (`[ИМЯ_СЕРВЕРА]`) и **перезагрузите**, если меняли имя.
4. Настройте **статический IPv4** и маску/шлюз/DNS так, чтобы с виртуальных/физических Windows 7 сервер **пинговался** по имени или по IP. Дальше — вручную то, что обычно требуется для «видимости» сервера в рабочей группе и для **ICMP (ping)** / **NetBIOS** с точки зрения брандмауэра.  
   - **Дополнительные параметры общего доступа:**  
     **Панель управления** → **Центр управления сетями и общим доступом** → слева **Изменить дополнительные параметры общего доступа**.  
     Для профиля, которым помечена ваша ЛВС (чаще всего **Частная**; при необходимости задайте отдельно **Гостевая** или **Все сети** — по политике безопасности), включите:  
     - **Включить сетевое обнаружение**;  
     - **Включить общий доступ к файлам и принтерам**.  
     Нажмите **Сохранить изменения**.  
   - **Профиль сети «Частная»:** если активное подключение к ЛВС помечено как **Общедоступная**, многие правила брандмауэра для обнаружения не сработают так, как ожидается. Переведите сеть в **Частную**: в том же **Центре управления сетями** нажмите на имя подключения → свойства (или ссылку **Изменить** рядом с профилем) → выберите **Частная сеть** / **Частный профиль** и сохраните. (Путь может чуть отличаться в зависимости от набора компонентов GUI.)
   - **Брандмауэр: группы правил для обнаружения и SMB:**  
     **Панель управления** → **Брандмауэр Защитника Windows** → **Дополнительные параметры** → раздел **Правила для входящих подключений**.  
     Отсортируйте по столбцу **Группа** (или пролистайте список) и для профиля **Частный** (и при необходимости **Доменный**) включите (**Включено** + **Разрешить** правило для всех строк этой группы, относящихся к вашему профилю):  
     1. Группа **Обнаружение сети** — на английской ОС: **Network Discovery** (несколько правил: NB-Name, UPnP, LLMNR и т. д.).  
     2. Группа **Общий доступ к файлам и принтерам** — на английской ОС: **File and Printer Sharing** (в том числе правило **Эхо-запрос ICMP** / *Echo Request ICMPv4-In*, если нужен **ping** с клиентов).  
   - **Службы (если после шагов выше сеть по-прежнему «пустая»):** `Win + R` → `services.msc` → найдите и для каждой (если служба установлена): тип запуска **Автоматически** или **Вручную**, затем **Запустить**:  
     - **Обнаружение SSDP** (*SSDP Discovery*);  
     - **Публикация ресурсов обнаружения функции** (*Function Discovery Resource Publication*);  
     - **Узел универсальных PNP-устройств** (*UPnP Device Host*).  
     На облегчённых установках Server часть служб может отсутствовать — такие строки пропускайте.  
   - Если DNS в сети нет, на **клиентах** Windows 7 можно прописать в `C:\Windows\System32\drivers\etc\hosts` строку вида `[IP_СЕРВЕРА] [ИМЯ_СЕРВЕРА]` (открыть Блокнот **от администратора**, сохранить без расширения `.txt`).
5. **Часовой пояс** и время — корректные (для журналов и возможных сертификатов).

**Интернет на сервере не используем** — обновления Windows и компоненты Store не рассчитываем; роли ставим с установочного носителя или заранее подготовленного `.iso` Server 2016 (см. шаг 3 при ошибке «не удаётся найти исходные файлы»).

## Шаг 1. Локальный пользователь для сеансов RDS

Создайте отдельную учётную запись для входа пользователей (не гостевую).

### 1.1. Ослабление требований к паролю (изолированная ЛВС, простой пароль)

По умолчанию Windows требует **сложный пароль** (длина, буквы разного регистра, цифры и т. д.). Чтобы разрешить, например, пароль `obsidian` для пользователя `obsidian`, измените **локальную политику безопасности** на сервере (в рабочей группе действует она, не доменные GPO).

1. Откройте **Локальную политику безопасности**: `Win + R` → введите `secpol.msc` → Enter.  
   (Тот же раздел доступен через `gpedit.msc`: **Конфигурация компьютера** → **Конфигурация Windows** → **Параметры безопасности** → **Политики учётных записей** → **Политика паролей**.)
2. В дереве слева: **Политики учётных записей** → **Политика паролей**. Справа откроется таблица «Политика» / **Параметр безопасности** — это текущие значения на сервере.
3. Дважды щёлкните строку **Пароль должен отвечать требованиям сложности** → выберите **Отключён** → **OK**.  
   **Проверка:** в колонке **Параметр безопасности** для этой политики должно стать **Отключен**, а не **Включен** (как на экране до изменения).
4. При необходимости ослабьте остальные ограничения, если мастер создания пользователя всё ещё ругается (имена в `secpol.msc` на русском):  
   - **Минимальная длина пароля** — например `0 знаков`;  
   - **Минимальный срок действия пароля** — `0 дн.`;  
   - **Вести журнал паролей** — `0 сохраненных паролей` (не хранить историю для запрета повторов);  
   - **Максимальный срок действия пароля** — для лаборатории часто задают **`0 дн.`** (пароль не истекает; если в вашей редакции «0» трактуется иначе — задайте большое число дней или снимите срок через свойства учётной записи пользователя в п. 1.2).
5. Закройте оснастку. Выполните обновление политики: командная строка от **Администратор** →  
   `gpupdate /force`  
   При странном поведении политики паролей — **перезагрузка** сервера.
6. (Необязательно) Проверка с командной строки администратора: **`net accounts`** — в выводе видны минимальная длина пароля, срок действия и т. п. (должны соответствовать тому, что вы задали в `secpol.msc`).

После этого задайте пароль пользователю (см. **п. 1.2 п. 3**). **Важно:** простые пароли допустимы только при осознанном риске (утечка из ЛВС, подбор). Для выхода в интернет или смешанных сетей используйте сложные пароли.

### 1.2. Создание пользователя и пароль

1. **Панель управления** → **Учётные записи пользователей** → **Учётные записи пользователей** → **Другой вид** → **Управление другой учётной записью** → **Создание учётной записи**  
   или: **Диспетчер серверов** → **Средства** → **Управление компьютером** → **Локальные пользователи и группы** → **Пользователи** → ПКМ → **Новый пользователь…**  
   Если пользователь `obsidian` уже создан — ПКМ по нему → **Задать пароль…** (или **Свойства** и смена пароля там, где доступно в вашей редакции).
2. Имя: например `[ПОЛЬЗОВАТЕЛЬ]` = `obsidian`.
3. Задайте **пароль** (после ослабления политики — по вашему правилу для ЛВС), снимите галку «Требовать смену пароля при следующем входе», если не хотите принудительной смены при первом входе; при необходимости отметьте «Срок действия пароля не ограничен».
4. Добавьте пользователя в группу **Пользователи удалённого рабочего стола** (обязательно для доступа по RDP после включения RDS):  
   **Управление компьютером** → **Локальные пользователи и группы** → **Группы** → **Пользователи удалённого рабочего стола** → **Добавить** → укажите `[ИМЯ_СЕРВЕРА]\[ПОЛЬЗОВАТЕЛЬ]`.
5. **Проверка членства в группе:** снова откройте свойства группы **Пользователи удалённого рабочего стола** и убедитесь, что в списке есть `[ИМЯ_СЕРВЕРА]\[ПОЛЬЗОВАТЕЛЬ]`. Либо в **командной строке от администратора**:  
   `net localgroup "Пользователи удалённого рабочего стола"`  
   (на сервере с английским интерфейсом имя группы: `Remote Desktop Users`).

**Особенность одной учётки на всех:** при одновременном входе нескольких человек под одним именем возможны ограничения политики «одна сессия на пользователя» и **риск порчи одного каталога хранилища** Obsidian, если все пишут в одно хранилище. Для «один активный редактор» это приемлемо; для нескольких одновременно — лучше разные учётки или разные хранилища.

## Шаг 2. Копирование Obsidian Portable на сервер

1. Создайте каталог, например `D:\RemoteApp\`.
2. Скопируйте папку `ObsidianPortable` в `D:\RemoteApp\ObsidianPortable` (полный путь к лаунчеру должен быть):  
   `D:\RemoteApp\ObsidianPortable\ObsidianPortable.exe`  
   Если файла `ObsidianPortable.exe` в корне нет (нестандартная сборка), точку входа уточняют по `ObsidianPortable\App\AppInfo\appinfo.ini` (ключ `Start=`).  
   **Несколько одновременных сеансов RDS с этой копией:** в `ObsidianPortable\App\AppInfo\Launcher\ObsidianPortable.ini` — **`SinglePortableAppInstance=false`** и **`CommandLineArguments`** с разным **`--user-data-dir=…%USERNAME%-%SESSIONNAME%`** (в репозитории уже так); иначе Electron не даст второй Obsidian. Подробнее — **шаг 5.2**.
3. Скопируйте папку `Инструкция` в `D:\RemoteApp\Инструкция`.
4. **Права NTFS на всю папку `ObsidianPortable` (иначе у пользователя не стартует).** Копирование под администратором обычно оставляет «только админы могут писать». Obsidian пишет журнал и данные в подкаталог **`Data`** (в т. ч. `Data\ObsidianAppData\obsidian.log`). Обычному пользователю нужно **изменение** каталога и наследование на вложенные папки и файлы.  
   - **Вручную:** ПКМ по `ObsidianPortable` → **Свойства** → вкладка **Безопасность** → **Изменить** → **Добавить** → введите, например, **`Прошедшие проверку подлинности`** (на английской ОС: *Authenticated Users*) → **Проверить имена** → OK → для этой группы отметьте **Изменение** (или **Полный доступ**, если политика организации это допускает). Нажмите **Дополнительно** → у выбранной записи **Тип** = **Разрешить**, **Применяется к:** **Этому каталогу, его подкаталогам и файлам** → **ОК** на всех окнах.  
   - **Командная строка от администратора** (путь подставьте свой; SID `*S-1-5-11` — «Прошедшие проверку подлинности», не зависит от языка ОС):  
     `icacls "D:\RemoteApp\ObsidianPortable" /grant *S-1-5-11:(OI)(CI)M /T`  
   В изолированной ЛВС это типичный компромисс для общей portable-копии; при жёстких требованиях безопасности вместо широкой группы добавляйте только учётки RDS и задавайте им **Изменение** на эту папку.  
   Если приложение уже стоит в **`C:\ObsidianPortable`**, либо выдайте такие же права на эту папку, либо **перенесите** дерево на `D:\RemoteApp\ObsidianPortable` и обновите путь в RemoteApp — так предсказуемее.

Проверка: с консоли сервера под **той же** учёткой `[ПОЛЬЗОВАТЕЛЬ]` **без** «Запуск от имени администратора» (войдите через смену пользователя или временно дайте ей локальный вход) запустите `ObsidianPortable.exe` и убедитесь, что приложение стартует **без** диалога JavaScript / `EPERM`. На этапе теста можно один раз войти под `[ПОЛЬЗОВАТЕЛЬ]` на консоль (если политика разрешает локальный вход).

## Шаг 3. Установка ролей Remote Desktop Services (узел сеансов)

Цель: установить **Remote Desktop Session Host**, чтобы пользователи подключались по RDP к **сеансу** на сервере (и дальше — к Obsidian). **Классическая публикация RemoteApp** через коллекции RDS в модели Microsoft обычно требует **Remote Desktop Connection Broker** и типично **домена**; в **рабочей группе** без брокера по [документации Microsoft](https://learn.microsoft.com/troubleshoot/windows-server/remote/install-rds-host-role-service-without-connection-broker) **не предоставляются** программы RemoteApp и веб-доступ RD Web — только удалённый **рабочий стол**. Обходные варианты (полный рабочий стол, `.rdp` с удалённым запуском приложения при разрешении политикой и т. п.) — см. **п. 3.7** и раздел **«Что дальше»**.

### 3.1. Запуск мастера

1. **Диспетчер серверов** → **Управление** → **Добавить роли и компоненты**.
2. **Тип установки:** установка ролей или компонентов.
3. **Выбор целевого сервера:** ваш локальный сервер.

### 3.2. Выбор ролей

1. На шаге мастера **«Роли сервера»** / **«Выбор ролей сервера»** установите флажок **«Службы удаленных рабочих столов»** (в английской локализации: **Remote Desktop Services**). Другие роли (DNS, IIS и т. п.) для минимального RemoteApp не обязательны.
2. Нажмите **«Далее»**. Если появится запрос **добавить необходимые компоненты** (зависимости роли) — выберите **«Добавить компоненты»** / согласитесь с предложением мастера.
3. На следующих шагах мастера дойдите до раздела **служб ролей** RDS — см. **п. 3.3**.

### 3.3. Службы ролей RDS

На шаге **службы ролей** обычно отмечают:

- **Remote Desktop Session Host** (Узел сеансов удалённых рабочих столов) — **обязательно** для пользовательских сеансов;
- **Remote Desktop Licensing** (Лицензирование удалённых рабочих столов) — **рекомендуется** на том же сервере, чтобы после льготного периода выдавать **RDS CAL** (в рабочей группе — в режиме **на устройство**, см. раздел **«Юридические и лицензионные особенности»**);
- **Remote Desktop Connection Broker** (Посредник подключений удалённых рабочих столов) и **Remote Desktop Web Access** — для **стандартного / быстрого** развёртывания RDS с **публикацией RemoteApp** через коллекции; в **рабочей группе** мастера в Диспетчере серверов часто **недоступны** (см. **п. 3.7**). Если вы сознательно ставите **только** узел сеансов + лицензирование **без** брокера, Microsoft прямо указывает: такая конфигурация **не открывает** доступ к **программам RemoteApp** и **RD Web** — только сеансы **полного рабочего стола** ([статья](https://learn.microsoft.com/troubleshoot/windows-server/remote/install-rds-host-role-service-without-connection-broker)).

Для **минимального** сценария в рабочей группе (сеанс рабочего стола + Obsidian на сервере) на первом проходе достаточно **Session Host** + **Licensing** (когда появятся CAL). Для **классического** RemoteApp в инфраструктуре Microsoft рассмотрите **домен** и развёртывание **с** Connection Broker — иначе опирайтесь на обходные способы в **п. 3.7**.

**Если RDS CAL пока нет:** узел сеансов всё равно можно установить — в течение **льготного периода (около 120 дней)** пользователи обычно могут подключаться **без** выданных CAL, пока вы разворачиваете лабораторию. Дальше для легальной работы нужны **сервер лицензирования RDS** (часто на том же сервере, если выбрали роль **Лицензирование**) и **пакеты RDS CAL** с ключами активации; иначе после истечения периода новые сеансы, как правило, **блокируются**. Для **только теста** в пределах 120 дней достаточно **Session Host**; роль **Лицензирование** имеет смысл ставить заранее, если планируете сразу готовить сервер к вводу CAL (без ключей она не «даёт» лицензии сама по себе).

### 3.4. Предупреждения мастера

- Мастер может сообщить, что для установки узла сеансов будут **удалены компоненты «лишних» ролей** (например, некоторые сценарии совместного Hyper-V и RDS) — на чистой VM это редко мешает; прочитайте текст и подтвердите.
- Может появиться требование **указать источник компонентов** — укажите букву диска с `.iso` Windows Server 2016 или папку `sources\sxs` с дистрибутива.

### 3.5. Завершение

1. Дождитесь окончания установки и **перезагрузите** сервер, если мастер потребует (часто дважды: после установки роли и после настройки).
2. После перезагрузки может открыться мастер настройки **Лицензирования RDS** — его можно пройти, когда будут готовы CAL. До настройки лицензирования узел сеансов опирается на **льготный период (~120 дней)**; по его окончании без валидных CAL подключения пользователей обычно прекращаются (см. раздел **«Юридические и лицензионные особенности»** выше и [RDS на Learn](https://learn.microsoft.com/windows-server/remote/remote-desktop-services/rds-client-access-license)).

### 3.6. Проверка

- **Диспетчер серверов** → **Средства** → **Remote Desktop Services** — пункты подменю (например **Диспетчер лицензирования удалённых рабочих столов**) должны открываться без ошибки «роль не установлена».
- В **Службах** (`services.msc`) должны быть запущены службы, связанные с **Remote Desktop Services** / **TermService**.

### 3.7. Рабочая группа: синее сообщение в Диспетчере серверов и отсутствие коллекций

Если на панели **«Общие сведения»** отображается текст вроде: *«Вы выполнили вход с учётной записью локального администратора. Чтобы управлять серверами и коллекциями, вам необходимо выполнить вход в качестве администратора домена»* — это **ожидаемо** для сервера в **рабочей группе**: обзор **Remote Desktop Services** и мастера **коллекций** в Диспетчере серверов рассчитаны на сценарий **Active Directory**.

1. **Это не означает**, что роли RDS «не установились». Проверка — по **Службам** и по возможности открыть **Средства** → **Диспетчер лицензирования удалённых рабочих столов** / **Средство диагностики лицензирования** под локальным администратором.
2. Microsoft для рабочей группы указывает: **инструменты Диспетчера серверов для RDS не работают** — управление узлом сеансов и лицензированием делают через **PowerShell** и локальные оснастки (пример настройки режима лицензирования и сервера лицензий — в [той же статье](https://learn.microsoft.com/troubleshoot/windows-server/remote/install-rds-host-role-service-without-connection-broker)).
3. **Практика без домена:** пользователи подключаются по RDP к **полному рабочему столу** сервера и запускают `ObsidianPortable.exe` с ярлыка; либо используют сценарий **шага 5** (файл **`.rdp`** с `remoteapplicationmode` / `remoteapplicationprogram` и политика **«Разрешать удаленный запуск любых программ»**).
4. Если нужен именно **классический** RemoteApp из коллекции — планируйте **домен** и развёртывание RDS **с** **Connection Broker** (и лицензирование по правилам Microsoft).

## Шаг 4. Брандмауэр для входящего RDP

1. **Панель управления** → **Брандмауэр Защитника Windows** → **Дополнительные параметры**.
2. **Правила для входящих подключений** — найдите правила **Удалённый рабочий стол — режим пользователя (TCP-In)** и **(UDP-In)** (названия могут слегка отличаться по языку ОС).
3. Убедитесь, что для нужного профиля (**Доменный** / **Частный** / **Публичный** — в зависимости от того, как помечен ваш сетевой профиль на сервере) правило **Включено** и **Разрешить**.
4. При необходимости создайте **отдельное правило** только для подсети клиентов (безопаснее, чем «весь мир»).

Порт по умолчанию: **TCP 3389** (и UDP для UDP-транспорта RDP в новых клиентах; для Win7 важен TCP).

## Шаг 5. Вариант А (рабочая группа, без брокера): удалённый запуск через `.rdp`

Сценарий **без** Connection Broker: клиент открывает **`.rdp`**, на узле сеансов запускается указанная программа. Нужны роли из шагов 3–4. Политика **«Разрешать удаленный запуск любых программ»** разрешает запуск по `.rdp` любого exe с сервера — раздавайте `.rdp` только доверенным сторонам.

### 5.1. Политика: разрешить удалённый запуск непубликованных программ

1. На **сервере** откройте локальный редактор политик: `Win + R` → `gpedit.msc` → Enter.
2. **Конфигурация компьютера** → **Административные шаблоны** → **Компоненты Windows** → **Службы удалённых рабочих столов** → **Узел сеансов удалённых рабочих столов** → **Подключения** (в интерфейсе бывает «удаленных» без **ё**).
3. **«Разрешать удаленный запуск любых программ»** (*Allow remote start of unlisted programs*) → **Включено** → **OK**.
4. `gpupdate /force` от администратора (при необходимости перезагрузка).

Без этой политики **Включено** запуск через `.rdp` с `remoteapplicationmode:i:1` часто запрещён.

### 5.2. Политика: один сеанс на пользователя или несколько

Тот же узел политик: **…Узел сеансов удалённых рабочих столов** → **Подключения**.

| Задача | Параметр в `gpedit.msc` (рус. Windows Server 2016) / ориентир EN | Обычное значение |
|--------|----------------------------------------------------------------|------------------|
| **Не больше одного** активного сеанса на одну учётную запись (типично для Obsidian и одного каталога данных) | **«Ограничить пользователей служб удаленных рабочих столов одним сеансом служб удаленных рабочих столов»** — *Restrict Remote Desktop Services user to a single Remote Desktop Services session* | **Включено** |
| **Разрешить несколько** одновременных сеансов одной и той же учётной записи (несколько клиентов под одним логином) | Тот же параметр | **Отключено** или **Не задана** |

После изменения — `gpupdate /force`.

**Несколько сеансов с одной копии portable:** в `ObsidianPortable\App\AppInfo\Launcher\ObsidianPortable.ini` — **`SinglePortableAppInstance=false`** и  
`CommandLineArguments=--user-data-dir="%PAL:DataDir%\ObsidianAppData-rds-%USERNAME%-%SESSIONNAME%"`  
(иначе второй сеанс с тем же каталогом данных Electron не стартует). Скопируйте `ObsidianPortable.ini` на сервер. В новой папке профиля **нет** `obsidian.json` из **[FileWrite]** — при первом запуске откройте vault вручную (например `D:\RemoteApp\ObsidianPortable\Data\Obsidian Vault`). Один vault на два активных сеанса — риск порчи данных.

### 5.3. Параметры файла `.rdp` (remoteapplicationprogram / remoteapplicationmode)

Минимально нужны строки (пути и имена подставьте свои):

- **`remoteapplicationmode:i:1`** — включить режим удалённого приложения (не полный рабочий стол как основной сценарий).
- **`remoteapplicationprogram:s:`** — **полный путь к `.exe` на сервере**, например `D:\RemoteApp\ObsidianPortable\ObsidianPortable.exe` (тот же путь, что в шаге 2).
- **`remoteapplicationname:s:`** — подпись в окне клиента (произвольная, например `Obsidian Portable`).
- **`shell working directory:s:`** — рабочий каталог на сервере (для portable обычно корень установки: `D:\RemoteApp\ObsidianPortable`).
- **`full address:s:`** — `[ИМЯ_СЕРВЕРА]` или `[IP_СЕРВЕРА]`.
- **`username:s:`** — по желанию подставить `[ПОЛЬЗОВАТЕЛЯ]` (иначе клиент спросит при подключении).

При необходимости: **`disableremoteappcapscheck:i:1`**. Точку входа в `.rdp` держите **`ObsidianPortable.exe`**, если не меняете схему portable.

### 5.4. Шаблон на диске

Файл **`Инструкция\Obsidian_RemoteApp_Workgroup.rdp`**: замените **`YOUR-SERVER-NAME`**, **`YOUR-USERNAME`**, при необходимости пути к exe и рабочему каталогу.

### 5.5. Клиент Windows 7

**SP1**, обновления, по возможности актуальный **RDP-клиент** (платформа RDP 8.x). Скопируйте `.rdp` на клиент, подключитесь; в рабочей группе домен часто пустой или имя сервера. Если открывается полный рабочий стол — п. 5.1, `remoteapplicationmode:i:1`, путь в `.rdp`, NTFS (шаг 2).

### 5.6. Проверка

Откройте `.rdp` с клиента или сервера под тестовым пользователем. При отказе — группа **Пользователи удалённого рабочего стола**, право **«Разрешить вход в систему через службы удалённых рабочих столов»** в `secpol.msc`, брандмауэр (шаг 4).

### 5.7. Лимиты времени сеанса (RemoteApp / отключённые сеансы)

См. [Microsoft: RemoteApp sessions disconnected](https://learn.microsoft.com/en-us/troubleshoot/windows-server/remote/remoteapp-sessions-disconnected).

`gpedit.msc` → **Конфигурация компьютера** → **Административные шаблоны** → **Компоненты Windows** → **Службы удалённых рабочих столов** → **Узел сеансов удалённых рабочих столов** → **Ограничение сеансов по времени**

| № | Политика (рус. Server 2016) | Значение |
|---|-----------------------------|----------|
| 1 | Задать ограничение по времени для отключенных сеансов | **Включено** → **1 минута** (или 2 мин, если 1 мин нет в списке) |
| 2 | Задать ограничение времени для активных, но бездействующих сеансов… | **Не задана** |
| 3 | Задать ограничение по времени для активных сеансов… | **Не задана** |
| 4 | Завершать сеанс при достижении ограничения по времени | **Включено** |
| 5 | Задать предел времени для выхода из сеансов RemoteApp | **Включено** → **1 минута** |

`gpupdate /force`. При нескольких сроках действует **меньший**. Диалог похожих сеансов на клиенте — по времени или снятие отключённых сеансов на сервере (админ). «Один сеанс на пользователя» — п. 5.2.

### 5.8. Сообщение PortableApps «не закрылся должным образом…»

Лаунчер PortableApps после обрыва сеанса, logoff по п. 5.7, сбоя или принудительного завершения процесса при следующем запуске может показать диалог про очистку и **сам завершится**. Нажмите **OK**, затем снова откройте `.rdp`.

### 5.9. Завершение Obsidian в чистом RemoteApp

Если нет рабочей команды выхода в интерфейсе (в т. ч. нет quit в палитре, **Ctrl+Q** / **Alt+F4** не закрывают приложение), в режиме **только RemoteApp** остаётся **закрыть окно** клиентом или **разорвать сеанс** — для лаунчера это часто **нестандартное** завершение, см. п. **5.8**. Аккуратно снять процесс можно из **полного** RDP-сеанса рабочего стола или с консоли сервера (**Диспетчер задач**).
